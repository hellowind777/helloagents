# 借鉴 Grill-me / Trellis 理念优化 HelloAGENTS 的方案（交互澄清模式）

> 源文章: Superpowers不再是最佳实践：2026年你该换个思路了 (烨笙, 2026-06-26)
> 评估日期: 2026-06-28
> 状态: 草案，待讨论

---

## 一、背景

文章核心论点：

1. **"做选择题 vs 写作文"** — AI 一次问一个问题让你拍板，比 AI 生成 2000 字方案让你逐条审查，心智负担低一个量级
2. **方法论对，包装要轻** — Superpowers 5 个大版本越做越重，用户关了又开，说明价值在但重量是问题。文章提出的 Grill-me（交互式一问一答）和 Trellis（结构化任务树）提供了更轻量的思路
3. **结构化任务树 + 精确验收标准** — Trellis 的每个任务有可独立验证的完成定义，长会话中不退化的持久真相源

HelloAGENTS 在结构化任务管理、上下文治理、质量闸门、跨平台、零依赖等方面已经做得很好，甚至超过文章所描述的工具。但在以下方面存在借鉴空间。

### 2.2 各命令是否需要交互澄清阶段

判断标准：**该命令是否涉及需求/设计决策，且这些决策在用户初始表述中通常不完整？**

| 命令 | 需要交互澄清？ | 理由 |
|------|----------------|------|
| `~plan` | **必须** | 方案包质量取决于需求理解精度。当前仅对"全新项目"启用交互模式，应扩展为所有场景默认 |
| `~prd` | **必须** | 当前 SKILL.md 第 87 行已有"每次只问一个问题，偏好选择题"，但执行是"AI 先给推荐，用户确认"——仍是批改作文。应改为：项目定位阶段 交互澄清确立范围 → 维度探索保持一问一答 |
| `~build` | **条件性** | 有方案包时按方案执行（不走交互澄清，决策已在 ~plan 中做完）。无方案包且 T2+ 任务时，自动重定向到 ~ask/~plan，不直接写代码 |
| `~ask` | **载体本身** | 升级后的 ~ask 就是 交互澄清模式的实现。~plan 和 ~prd 的 交互澄清阶段本质上复用此交互模式 |
| `~auto` | 不需要 | 元路由器，交互澄清委托给下游命令（~ask/~plan/~prd），自身不做需求澄清 |
| `~qa` | 不需要 | 验证已有产物，输入是代码和方案包，无设计决策 |
| `~test` | 不需要 | 范围明确（"给 X 写测试"），无歧义 |
| `~loop` | 不需要 | 长任务包装器，委托给 /goal → ~auto → ~qa |
| `~init` | 不需要 | 机械操作（创建目录/文件/知识库），无设计决策 |
| `~commit` | 不需要 | 机械 git 操作 |
| `~clean` | 不需要 | 机械清理 |
| `~help` | 不需要 | 信息展示 |

### 2.3 `~office` 是否应独立存在

**对比 `~idea` 和 `~office`：**

| 维度 | `~idea` (当前) | `~office` (当前) |
|------|-------------|---------------|
| 核心问题 | "有哪些做法？" | "该不该做？做多大？" |
| 给方案 | 2-4 个方向 | 3 档范围（不做/最小/标准） |
| 挑战前提 | 不显式挑战 | 显式挑战 1-3 个前提 |
| 给推荐 | 推荐一个方向 | 落到 4 种判断之一 |
| 输出 | 方向+差异+推荐+升级路径 | 结论+判断依据+升级路径 |
| 创建文件 | 不 | 不 |
| 升级到 | ~office/~plan/~build/~prd/~auto | ~idea/~plan/~build/~prd/~auto |

两者骨架几乎相同：纯对话 → 多方案 → 给推荐 → 升级路径。差异极其微妙且在实际对话中无法隔离——当用户说"我想做X"，方向探索和范围判断是**同一次对话中自然交替出现的**：

1. "你主要想解决什么问题？"（动机理解）
2. "两个方向：A 改动小但扩展性差，B 需重构但长期收益大——你倾向哪个？"（方向对比 = ~idea 领域）
3. "考虑到当前约束，建议先做最小切口验证。如果反馈好再扩展。你觉得呢？"（范围收缩 = ~office 领域）
4. "我怀疑一个前提：你们是否真的有这么多并发用户需要这个优化？能确认吗？"（前提挑战 = ~office 领域）

**这是一次连续对话，不是一个命令切换。**

**结论：`~office` 合并入 `~ask`。** 交互澄清中自然覆盖方向探索、价值判断、范围收缩、前提挑战。用户只需记住"不清楚做什么 → `~ask`"这一个入口。

## 三、`~idea` 命名评估 → 建议更名为 `~ask`

### 3.1 评估方法

从 6 个维度对 12 个候选名称交叉打分（每维 1-5 分，满分 30）：
- **语义准确性**：名称是否准确传达"AI 通过一问一答帮我理清需求"？
- **易记性**：是否简短？是否容易和命令行为建立联想？
- **易读性**：对中学英语水平（~1500 词汇量）的用户是否常见？
- **区分度**：是否与现有命令混淆？
- **一致性**：是否符合现有命令体系的命名风格（动词优先，小写）？
- **行业惯例**：同类工具是否有类似命名？

### 3.2 评分结果

| 排名 | 候选 | 得分 | 核心优势 | 核心弱点 |
|------|------|------|---------|---------|
| **1** | **`~ask`** | **27/30** | 3 字母；小学词汇；动词；最准确描述 交互澄清一问一答的核心行为；"不清楚要做什么 → 让 AI 来问我 → ~ask" | 无显著弱点 |
| 2 | `~start` | 26/30 | 小学词汇；"不知道从哪开始 → ~start" 天然入口联想 | 未传达 Q&A 特性；可能与 ~init 混淆 |
| 3 | `~try` | 24/30 | 3 字母；小学词汇；鼓励探索 | 未传达结构化澄清 |
| 4 | `~talk` | 23/30 | 小学词汇；传达对话感 | 过于泛化 |
| 5 | `~focus` | 23/30 | 传达"聚焦/收敛" | 中学以上词汇 |
| 6 | `~guide` | 22/30 | 传达"引导过程" | 5 字母；中学以上词汇 |
| 7 | `~aim` | 22/30 | 3 字母；传达"瞄准目标" | 中学以上词汇 |
| 8 | `~tell` | 21/30 | 小学词汇 | 单向（"我告诉你"），实际是 AI 问我 |
| 9 | `~pick` | 20/30 | 传达"做选择" | 只覆盖选择题部分，遗漏澄清过程 |
| 10 | `~what` | 20/30 | 小学词汇 | 疑问词，破坏动词命名一致性 |
| 11 | `~how` | 20/30 | 3 字母 | 同上 |
| 12 | `~chat` | 18/30 | 常见词 | 太空泛；所有 AI 交互都是 chat |
| — | `~idea` (当前) | **15/30** | 中学词汇，4 字母 | 名词（非动词）；"点子"语义与新定位严重偏离；描述的是**产出**而非**过程** |

### 3.3 结论

**`~idea` 更名为 `~ask`。** 核心理由：

1. Grill-me 的核心是**过程**（结构化一问一答），不是**产出**（点子/想法）。`~idea` 是名词，描述"你得到什么"；`~ask` 是动词，描述"你们一起做什么"
2. `~ask` 3 字母，是英语中最基础的 100 词之一，任何中学水平的人都知道
3. 命令体系自然对齐：`~ask`（问清楚）→ `~plan`（做方案）→ `~build`（写代码），三个层级递进
4. 心智模型明确："我不确定要做什么 → 让 AI 来问我 → `~ask`"

### 3.4 向后兼容

保留 `~idea` 作为兼容别名（与现有 `~do`→`~build`、`~design`→`~plan`、`~review`→`~qa` 一致）：
- 用户输入 `~idea` 等同于 `~ask`，行为完全一致
- 1-2 个大版本后逐步废弃

### 3.5 迁移范围

共 16 个文件约 35 处引用需要更新：

| 类别 | 文件 | 改动 |
|------|------|------|
| 目录重命名 | `skills/commands/idea/` → `skills/commands/ask/` | 目录 + SKILL.md frontmatter |
| 命令交叉引用 | `auto/SKILL.md`(5处), `office/SKILL.md`(1处), `help/SKILL.md`(1处), `helloagents/SKILL.md`(1处) | 更新命令名 |
| Bootstrap | `bootstrap.md`(5处), `bootstrap-lite.md`(2处) | 更新命令名 |
| README | `README.md`(4处), `README_CN.md`(4处) | 更新命令名 |

| 测试 | `runtime-route.test.mjs`(5处), `workflow-runtime-contract.test.mjs`(1处), `workflow-skill-contract.test.mjs`(1处) | 更新路径和断言 |

## 四、最终命令架构

```
用户入口层（2个）：
  ~ask     对话式需求澄清（交互澄清），纯对话不创建文件
           自然覆盖：方向探索 + 价值判断 + 范围收缩 + 前提挑战
           输出：结构化选择摘要 + 升级路径建议
           兼容别名：~idea
  ~auto    万能自动执行入口，自动选路并持续执行
           内部委托给 ~ask/~plan/~prd/~build/~qa，遇到歧义走交互澄清

规划层（2个）：
  ~plan    先走 交互澄清（~10分钟）→ 生成方案包（requirements/plan/tasks/contract）
           文档是交互结果的"记录"，而非"发现过程"
           快速通道：用户说"直接做，不用问我"时跳过交互
  ~prd     先走交互澄清 确立范围 → 逐维度深入探索（保持一问一答）→ 生成完整 PRD + 方案包
           已是最接近 Grill-me 的命令，改动最小

执行层（1个）：
  ~build   有方案包 → 按方案包执行
           无方案包 + T2+ → 自动重定向到 ~ask / ~plan
           无方案包 + T1（如"加个loading"）→ 直接执行

质量层（2个）：
  ~qa      统一质量闭环：审查 → 验证命令 → 修复循环 → 证据
  ~test    为指定模块编写测试

专项层（4个）：
  ~loop    长任务入口（/goal → ~auto → ~qa）
  ~init    项目初始化 + 知识库同步
  ~commit  规范化 git 提交
  ~clean   清理 + 归档

元命令（1个）：
  ~help    帮助信息

淘汰（1个）：
  ~office  合并入 ~ask
```

**命令数：13 → 12。用户心智路径：**

- "不清楚要做什么" → `~ask`（一问一答厘清）
- "清楚要做什么，需要方案" → `~plan`（交互澄清确认 → 生成方案包）
- "直接推进，不想选" → `~auto`（自动选路，遇歧义走交互澄清）

## 四、具体改进项（共 4 项）

### 改进 A：`~ask` — 交互澄清载体（原 `~idea` 升级 + 吸收 `~office`）

**涉及文件**：`skills/commands/ask/SKILL.md`（原 `skills/commands/idea/SKILL.md`）

**改动内容**：

1. 一问一答交互模式：每次只问一个问题，带推荐选项，选择题优先
2. AI 主动读代码库寻找答案，减少对用户的打扰
3. 对话中自然覆盖：方向探索（原 ~idea）+ 价值判断/范围收缩/前提挑战（原 ~office）
4. 不创建任何文件，不写方案包
5. 结束时输出结构化选择摘要（非方案文档）
6. 保留升级路径建议
7. 用户说"直接做，不用问我"时跳过交互，直接输出推荐方向

---

### 改进 B：`~plan` 和 `~prd` 前置 交互澄清阶段

**涉及文件**：`skills/commands/plan/SKILL.md`, `skills/commands/prd/SKILL.md`

**`~plan` 改动**：

1. 第 2 步"需求澄清"：将"交互模式"从"全新项目专用"提升为**所有场景默认**
2. 已有代码库时，AI 先静默读代码，然后以 一问一答选择题形式逐条确认理解，而非生成方案让用户审查
3. 用户说"直接做，不用问我"时走"假设模式"快速通道
4. 交互确认后才进入方案生成（第 4-5 步）——文档变成记录而非发现过程

**`~prd` 改动**：

1. 第 2 步"项目定位"改为 交互澄清：逐条确认项目类型、核心范围、关键约束
2. 第 3 步"维度探索"保持一问一答模式，且改为"AI 先提问"而非"AI 先给推荐让用户确认"
3. 用户说"默认"时 AI 按推荐快速填充；说"展开"时深入讨论

---

### 改进 C：`~build` 无方案包时的重定向逻辑

**涉及文件**：`skills/commands/build/SKILL.md`

**改动内容**：

1. 第 2 步"需求与范围确认"增加判断：
   - 有活跃方案包 → 按方案包执行（当前行为，不变）
   - 无方案包 + 任务可判定为 T1（如"加个 loading 动画""修个 typo"）→ 直接执行
   - 无方案包 + 任务 T2+ → **不直接写代码，自动重定向到 ~ask 或 ~plan**
2. 这避免了用户说 `~build "做个登录系统"` 时 AI 凭空猜测导致返工

---

### 改进 D：任务完成标准从"可量化"修正为"可独立验证"

**涉及文件**：`templates/plans/tasks.md`, `skills/commands/plan/SKILL.md`, `skills/qa-review/SKILL.md`

**核心原则**：不是"必须写数字"，而是"必须能独立判定真假"——任何人（人或 AI）读完完成标准后，能**无需额外上下文**判断任务是否真的完成。

**不同任务类型的验证方式**：

| 任务类型 | 错误写法 | 正确写法 |
|---------|---------|---------|
| API 端点 | "实现登录接口" | "POST /login 传入有效凭据返回 200 + token；无效凭据返回 401" |
| 文档 | "写好说明文档" | "README 覆盖安装/使用/API 三节；代码示例可在本地执行" |
| 重构 | "优化性能" | "现有测试全绿；公开 API 签名不变；lint 零告警" |
| 调研 | "调研一下方案" | "报告回答 3 个问题：可行性/成本/风险，每条附来源" |
| UI 组件 | "做个按钮" | "按钮在 hover/focus/active/disabled 四种状态下视觉正确；点击触发 onClick" |
| Bug 修复 | "修好这个 bug" | "复现步骤跑通后不再触发错误；加一条回归测试" |

**具体改动**：

1. `templates/plans/tasks.md` 模板，完成标准提示语改为：
   ```
   完成标准：可独立验证的布尔条件——任何人读完就能判断任务是否完成。用 Given-When-Then
   或等价形式。不写"功能正常""符合预期"等无法独立判断的表述。
   ```
2. `~plan` SKILL.md 第 4 步（方案细化）：增加约束——每个任务的完成标准必须能由 `~qa` 命令独立验证，不依赖主会话上下文
3. `qa-review` SKILL.md：任务验证环节增加逐条核对完成标准的步骤

### 改进 E：Bootstrap 维护原则（替代原"表达效率优化"）

**评审意见**：对原方案四个子项逐一审查后，做如下裁定。

**1. "压缩冗余" → 取消**

当前 bootstrap 规则已经非常精炼，进一步压缩存在歧义风险。规则的精确性优先于简短性。如果压缩导致 AI 对规则产生不同解读，节省的 token 远不能弥补行为偏差带来的返工成本。因此：**不做系统性压缩**。仅当某条规则被实际观察到与另一条完全语义重叠时，才做合并——且必须保证合并后语义完全等价。

**2. "结构化替代段落" → 不做强制要求**

长段落 vs 结构化列表各有利弊：
- 列表形式可能增加行数（每条一个 bullet），未必节省 token
- 某些规则需要段落中的因果连接词来表达"为什么"和"否则会怎样"，拆成列表后可能丢失这种逻辑链条
- AI 对段落的深层理解在复杂规则上可能优于对孤立列表项的理解

因此：**不强制段落改列表**。是否使用列表由规则本身的表达需求决定——适合列表的就用列表（如枚举条件），适合段落的就用段落（如行为哲学）。清晰度优先于格式一致性。

**3. "渐进式分层" → 取消**

当前 bootstrap 中所有规则适用于所有场景，不存在"某些规则只在特定场景需要"的情况。分层注入只会增加 runtime 侧复杂度，且无法减少实际的 token 消耗（因为所有规则最终都要加载）。如果某条规则真的只在特定场景适用，它本来就应该在对应的 hello-* skill 中，而不是在 bootstrap 里。

**4. "Token 预算机制" → 修正为"质量优先原则"**

原机制"增量为零时才能往 bootstrap 加内容"存在方向性错误：**产出质量优先于 token 预算**。如果一条新规则能预防一个已知的 AI 行为偏差，它值得加入 bootstrap，不管它消耗多少 token。

修正后的原则：
- **驱动力**：bootstrap 改动必须由**实际观察到的 AI 行为偏差**驱动，而非美学或结构偏好
- **判定标准**：加入新规则前，必须有证据表明没有这条规则时 AI 会在某场景下犯某类错误
- **审视现有**：定期审视 bootstrap 中是否有规则已经因模型能力提升而不再需要——这是唯一合理的"瘦身"场景
- **不设 token 硬上限**：规则精准性永远高于 token 效率

**总结**：Bootstrap 维护的核心从"表达效率优化"修正为**精准维护**——不主动压缩、不强制格式、不分层注入、不设 token 硬上限。改动的唯一驱动力是**实际观察到的行为偏差**。

## 五、不做的事项

| 事项 | 理由 |
|------|------|
| 缩减 Guard / Gate 系统 | 安全闸门和质量闸门是 HelloAGENTS 的核心差异化价值 |
| 新增 `~clarify` 命令 | 用 `~ask` 覆盖此场景，避免命令膨胀 |
| Cursor 宿主支持 | 明确不需要 |
| Bootstrap 大规模重写/删减 | 规则是必要的，风险过高 |
| 缩减为极简 prompt | 多层质量保障是 HelloAGENTS 存在的理由 |
| `dangerously-skip-permissions` 作为常规模式 | 安全不可妥协；仅作为 `~auto` 下可选的信任模式 |
