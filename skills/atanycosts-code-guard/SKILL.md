# Atanycosts Code Guard

在当前 HelloAGENTS 工作流中，这个 skill 负责 **代码完成后的个人质量守门**。
它不替代 `hello-review` 或 `hello-verify`，而是在它们基础上补一层更严格的 Atanycosts 标准。

## 何时激活

- 完成代码文件的创建或修改
- 用户显式调用 `/atanycosts-code-guard`
- 用户要求“审查 / 守门 / check / 看看这些改动有没有不符合”

## 守门目标

确保改动符合这三条硬规则：

1. **明确性优先于隐性容错**
2. **配置化优先于硬编码**
3. **精准处理优先于笼统兜底**

## 审查维度

### 1. 防御性编程与静默兜底

判违规：

- `catch (Exception e)`、`except Exception`、`catch (Throwable)`
- 空 catch、`try/except: pass`
- 非法输入直接返回空值、默认值、假成功
- 新逻辑失败时偷偷 fallback 到旧逻辑

修复方向：

- 用精准异常类型
- 非法输入做前置校验并抛出带信息的具体异常
- 无法处理的异常完整向上透传
- 删除“失败时偷偷给个能跑结果”的补丁逻辑

### 2. 硬编码

判违规：

- 魔法数字、写死阈值、写死重试次数、写死分页大小
- 固定路径、固定 URL、固定本地目录
- 明文账号、密码、Token、连接串

例外：

- `0 / 1 / -1`
- 常见单位换算值
- 单元测试断言值

修复方向：

- 提取命名常量
- 改函数参数
- 改配置文件或运行时配置

### 3. 伪需求代码

判违规：

- 旧方法和新方法并存
- `legacyMode`、`useNewApi`、`temp_`、`_deprecated`
- `// 修改后`、`// removed`、`// old code`
- 保留未使用代码只为“以后可能还用”

修复方向：

- 二选一，只保留当前语义
- 删除未使用代码和过渡注释
- 删除兼容 flag 与补丁包装

### 4. 环境与配置违例

判违规：

- Java `application.yml` 写 `${ENV_VAR:default}`
- 前端写死后端 URL
- 代码里直接带数据库账号、密码或密钥

修复方向：

- Java 配置保持纯静态值
- 前端 API 基址走运行时配置
- 敏感信息移出源码

### 5. 文案口径

判违规：

- 营销词、夸张词、AI 味太重的注释或 UI 文案
- 单纯复述代码行为的注释
- 面向用户暴露接口、状态码、堆栈或内部字段名

修复方向：

- 注释只解释 WHY，不解释显而易见的 WHAT
- UI 文案改为工程化、客观、可执行表述
- 屏蔽技术细节泄露

## 审查方式

- 如果用户要的是 **review / check**，先给出按严重度排序的发现，带文件定位与理由
- 如果用户已经授权你 **直接实现 / 直接修**，发现明显违例时可以顺手修掉，再在收尾里说明
- 涉及前端时，与 `atanycosts-frontend-guard` 和 `hello-ui` 协同
- 涉及重构尾巴时，交给 `atanycosts-refactor-clean`

## 额外要求

- 行为语义变更时，记得同步 `.helloagents/` 中相关知识、方案或文档
- 验证优先看外部可观察行为，不靠主观“应该没问题”
- 不为了“凑一次提交”放松 runtime contract 或 SSOT 约束
