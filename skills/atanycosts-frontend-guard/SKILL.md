# Atanycosts 前端工程化守门

这个 skill 的定位不是替代 `hello-ui`，而是在现有 UI 质量基线之上，再补一层 **工程化落地守门**。

## 何时激活

- 任务涉及前端 UI / UX / 页面布局 / 组件交互
- 任务容易踩到导航、侧边栏、表格、弹窗、长列表、复杂表单这些工程细节
- 用户显式调用 `/atanycosts-frontend-guard`

## 与 hello-ui 的关系

- `hello-ui`：负责设计简报、设计契约、视觉方向、实现映射与视觉验收
- `atanycosts-frontend-guard`：负责工程化门禁、状态覆盖、滚动与布局稳定性、文案与数据呈现口径

推荐顺序：

1. 先按 `hello-ui` 明确设计方向
2. 再用本 skill 做工程化自检
3. 收尾时保留最小可执行手动验收步骤

## 设计口径

### 1. 避免模板化审美

禁止默认 AI 味页面：

- 紫白渐变默认主题
- Hero + 三卡片模板化首屏
- 只有居中排版、没有层级
- emoji 充当功能图标
- 技术术语直接暴露给终端用户

### 2. 文案口径

用户可见文本要求：

- 说人话，不说接口话
- 给出原因和下一步，不只报“失败”
- 按钮文案用动作词
- 禁止泄露状态码、字段名、堆栈和开发术语

### 3. 数据呈现

- 表头、标签和状态值必须翻译成业务语义
- 时间、金额、百分比必须格式化
- 空值统一成清晰占位
- 不把 UUID、数据库 ID 当主展示内容

## 工程化门禁

### A. 布局与滚动

阻断项：

- 父子滚动链缺 `min-height: 0`
- 长文本把 flex / grid 撑爆
- 页面和内容区双滚动条打架
- Modal / Drawer 打开后背景还跟着滚
- 侧边栏不是固定或 sticky，滚一下就跑了

### B. 文本与排版

阻断项：

- 超长文本、URL、路径没有断行策略
- 标题或摘要没有截断与完整内容入口
- 字体体系混乱
- 多语言场景按固定宽度拍脑袋布局

### C. 响应式

默认 PC 断点至少检查：

- `1024px`
- `1280px`
- `1920px`

阻断项：

- 小屏布局挤爆
- 大屏内容过空、没有 `max-width`
- 侧边栏折叠后不可用

### D. 表单

阻断项：

- 必填字段无统一标识
- 输入时就疯狂报错
- 提交失败找不到第一个错误字段
- 提交按钮不防重
- 离开页面时未保存修改直接丢失

### E. 表格与列表

阻断项：

- 列宽无约束，内容挤爆整表
- 空表格完全空白，没有空状态
- 没有 loading / empty / error / permission 状态区分

## 状态覆盖

任何中高复杂度页面至少检查：

- loading
- empty
- error
- success
- disabled
- danger
- permission denied

优先级：

`loading > error > empty > permission > normal`

## 实施要求

- 写前端时同时想清楚状态和滚动，不要等 UI “差不多了”再补
- 视觉方向以 `hello-ui` 为准，工程细节以本 skill 为准
- 如果发现后端语义不稳定、接口字段暴露过重或文案不适合直接展示，要反推调用链一起修
- 用户要求“深度看看分析功能有没有问题”这类审查时，前端文案、状态、数据口径和交互路径都要算在审查范围里
